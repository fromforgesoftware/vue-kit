# WeekdaysPicker

Multi-select picker for days of the week. Renders seven round day buttons in the user-preferred week-start order with roving focus, full keyboard navigation, and `aria-checked` semantics.

## When to use

- Configure a recurring schedule by weekday (working days, rota patterns, shift availability).
- Filter records by day-of-week (timetable views, attendance summaries).
- Compact read-only display of a weekday set inside table rows or summary cards (use `readonly`).

## When not to use

- Picking calendar dates — use [Calendar](?path=/docs/dates-calendar--docs) or [DateField](?path=/docs/dates-datefield--docs).
- Picking a single recurring weekday — a [RadioGroup](?path=/docs/form-radio--docs) or [Select](?path=/docs/form-select--docs) reads more clearly when only one choice is allowed.
- More than seven options — use a multi-select [Listbox](?path=/docs/form-listbox--docs).

## Anatomy

<Canvas of={WeekdaysPickerStories.Default} />

A `<fieldset>` with a visually hidden `<legend>` wraps seven `<button role="checkbox">` elements. The button order follows the user's `getWeekStart()` preference.

`min-selection` enforces a floor on the selected count — useful when an empty selection is invalid (working-day patterns require at least one day). `error` flips unselected days to a destructive border, sets `aria-invalid="true"`, and shifts the focus ring to the destructive tone. Pair it with `describedBy` pointing to your message id.

## Accessibility (WCAG 2.2 AA)

- **4.1.2 Name, role, value**: rendered as a `<fieldset role="group">` with a hidden `<legend>` and a configurable `label` (also reflected as `aria-label`). Each button is `role="checkbox"` with `aria-checked` and `aria-label` set to the full day name.
- **2.4.6 Headings and labels**: `label` should describe the selection's purpose (e.g. "Working days" rather than just "Days").
- **2.5.8 Target size**: every size's painted hit area is ≥ 24 × 24 CSS px.
- **2.4.7 Focus visible**: focused day draws a 2 px ring at ≥ 3 : 1 contrast against the surrounding surface.
- **2.1.1 Keyboard**: full Arrow / Home / End / Space / Enter model managed by the component's roving-focus implementation.
- **3.3.1 Error identification**: pass `error` and `describedBy` together so the message is announced when the user lands on the field.

## Responsive behaviour

<Canvas of={WeekdaysPickerStories.InteractiveResponsive} />

The picker is `flex flex-wrap` — at very narrow viewports (< 320 px wide) days wrap onto multiple lines while keeping their hit area. For row-based table cells use `size="sm"` to keep height compact.

## Do / Don't

<table>
	<thead>
		<tr>
			<th>Do</th>
			<th>Don't</th>
		</tr>
	</thead>
	<tbody>
		<tr>
			<td>
				Pass a descriptive <code>label</code> (e.g. <code>"Working days"</code>).
			</td>
			<td>Rely on a visual heading alone — the fieldset still needs a programmatic name.</td>
		</tr>
		<tr>
			<td>
				Use <code>min-selection</code> when an empty set is invalid for the domain.
			</td>
			<td>Validate empty selection only on submit — the user has no per-click signal.</td>
		</tr>
		<tr>
			<td>
				Use <code>error</code> + <code>describedBy</code> for validation messages.
			</td>
			<td>Toggle to a manually styled wrapper — the error tone should travel with the field.</td>
		</tr>
		<tr>
			<td>
				Use <code>readonly</code> for table-cell display so users can still focus and copy.
			</td>
			<td>
				Use <code>disabled</code> for read-only display — disabled removes accessibility
				affordances.
			</td>
		</tr>
		<tr>
			<td>
				Match <code>size</code> to the surrounding form rhythm.
			</td>
			<td>
				Override <code>class="size-..."</code> on the buttons — use the <code>size</code> variant.
			</td>
		</tr>
	</tbody>
