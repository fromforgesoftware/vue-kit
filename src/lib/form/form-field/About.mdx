# FormField

Pairs a Label, a control, optional helper text, and an error message in one accessible compound. FormField owns the field id, `aria-describedby` wiring, and the required indicator — consumers pass any control via the default slot and bind a scoped-slot payload for accessibility hooks.

## When to use

- Any form input that needs a label, helper text, or validation messaging.
- Settings-page rows where a label + description stand on the left and a Switch / Select sits on the right.

## When not to use

- A label-less affordance (icon-only search bar) — pass `aria-label` directly to the control.
- A purely decorative input pair — FormField is for accessibility-wired controls.

## Composition

Pass the control via the default slot and bind the scoped-slot props for accessibility wiring:

```vue
<FormField label="Email" :error="errors.email" required>
  <template #default="{ id, error, required, ariaDescribedBy }">
    <Input
      :id="id"
      v-model="email"
      :error="error"
      :required="required"
      :described-by="ariaDescribedBy"
      type="email"
    />
  </template>
</FormField>
```

## Keyboard interaction

FormField doesn't add keyboard handling of its own — keystrokes flow to the slotted control. See the documentation of the control for details.

## Accessibility (WCAG 2.2 AA)

- **2.4.6 Headings and labels**: every FormField has a programmatic `<label>` paired with the slotted control via `for` / `id`.
- **3.3.1 Error identification**: when `error` is set, the message is rendered with `role="alert"` and the control receives `aria-invalid="true"` via the scoped slot.
- **3.3.2 Labels or instructions**: the description is rendered with a stable id and exposed through `aria-describedby` so screen readers read it after the label.
- **3.3.3 Error suggestion**: if you can suggest a fix, include it in the error message — FormField does not generate suggestions on its own.
- **2.4.7 Focus visible**: focus styling lives on the slotted control; FormField never absorbs the focus ring.

## Responsive behaviour

Vertical layout has no breakpoint behaviour. Horizontal layout stacks at mobile and switches to row at md+ (768 px).

## Do / Don't

	<tbody>
		<tr>
			<td>
				Bind the scoped-slot props (<code>id</code>, <code>error</code>,{' '}
				<code>ariaDescribedBy</code>) on the control.
			</td>
			<td>
				Set the control's <code>id</code> by hand and let it drift from the label's <code>for</code>
				.
			</td>
		</tr>
		<tr>
			<td>Replace the description with the error when validation fails.</td>
			<td>Show both at once — pick the one that matters now.</td>
		</tr>
		<tr>
			<td>
				Use <code>layout="horizontal"</code> for settings pages with a divide-y parent.
			</td>
			<td>Use horizontal for short forms with three or four fields — vertical reads better.</td>
		</tr>
		<tr>
			<td>
				Use a real label and only hide it with <code>srOnlyLabel</code> when needed.
			</td>
			<td>Replace a label with a placeholder — placeholders disappear on input.</td>
		</tr>
	</tbody>
