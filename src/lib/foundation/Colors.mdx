# Colors

Every colour in the library is defined as an [OKLCH](https://oklch.com) triplet stored on a CSS custom property. Components reference these via the `color-*` Tailwind utilities (`bg-primary`, `text-foreground`, `border-input`, etc.) — never via hex codes or hard-coded `oklch()`.

OKLCH is perceptually uniform: a 0.1 step in lightness is roughly the same visual distance everywhere on the spectrum, which makes WCAG contrast and dark-theme inversion easier to reason about than HSL or sRGB.

## Why this matters

- **Themability**: a single `--primary` change cascades across every component.
- **Dark mode**: only the `--*` raw values change in `[data-theme='dark']`; the `--color-*` aliases stay constant.
- **WCAG**: contrast ratios are testable from the design tokens alone — axe verifies them automatically per story.

### Soft variants

Solid colour tokens have a "soft" companion built on `/10` and `/20` opacity modifiers — these are used by the `soft-*` Badge variants and any low-emphasis status display. The pattern: tinted background (`bg-success/10`) with full-strength foreground (`text-success`).

## Surface tokens (specialised)

Two namespaces extend the semantic palette for specialised patterns:

- **`color-sidebar-*`** — the navigation chrome. Separate from `card` so sidebar can feel rooted while content cards float.
- **`color-chart-*`** (1 – 5) — categorical chart data series. Picked for perceptual distinctness, not brand alignment.
- **`color-st-*`** (Schedule Timeline) and **`color-ec-*`** (Event Calendar) — pattern-specific category palettes (work, leave, meeting…). Don't reuse them in unrelated contexts; they carry domain meaning.

## Theming

The default light theme lives in `[data-theme='light']` (and `:root`). A dark theme can be introduced by adding a `[data-theme='dark']` block in `_colors.css` and toggling the attribute on `<html>`. **No component should reference `dark:` Tailwind variants directly** — colour inversion is handled by the token swap.

## Contrast & a11y

Every solid `*-foreground` pairing meets **WCAG 1.4.3 (4.5 : 1)** against its base, verified by axe per story. Soft variants meet contrast against their tinted background (`bg-success/10 + text-success` ≈ 7 : 1).

When introducing a new variant, run the rendered story through the `Accessibility` panel. If contrast fails, adjust the OKLCH lightness — don't fall back to `text-white` or `text-black` literals.

## Don't

- Hard-code `oklch(...)` values in component CSS or stories. Use the token.
- Use `dark:` variants. Theme via attribute swap.
- Introduce new chart colours. Pick from `chart-1` … `chart-5`.
- Use semantic tokens decoratively (`bg-success` for visual interest). They carry meaning to assistive tech and the design system.
