# Alert Dialog

A modal confirmation that interrupts the user with a binary decision. Built on Reka UI â€” has the `role="alertdialog"` ARIA role, **ignores** Escape and overlay click, and forces a deliberate Cancel / Action choice. Use for irreversible actions.

## When to use

- Deleting an entity (workspace, account, file).
- Logging out with unsaved changes.
- Any action where the user might lose data or hit a billable side-effect.

## When not to use

- For routine forms or rich content â€” use [Dialog](?path=/docs/general-dialog--docs).
- For "are you sure" prompts on safe, reversible actions â€” they're noise.
- For info-only acknowledgements â€” use a [Toast](?path=/docs/general-toast--docs).

## Type-to-confirm

<Canvas of={AlertDialogStories.DestructiveConfirm} />

For high-blast-radius deletions, set `variant="destructive-confirm"` on the root and pass `confirmValue` to `AlertDialogContent`. The dialog renders a labelled input and disables the Action button until the user types the value exactly.

## Accessibility (WCAG 2.2 AA)

- **2.4.3 Focus order**: focus moves to Cancel on open, is trapped while open, and returns to the trigger on close.
- **3.2.2 On input**: only Cancel / Action close the dialog â€” no accidental dismissals from outside taps or Escape.
- **4.1.2 Name, Role, Value**: the panel has <code>role="alertdialog"</code>, with `aria-labelledby` and `aria-describedby` wired by the title and description.
- **2.4.7 Focus visible**: each footer button keeps the standard `:focus-visible` ring.
- **3.3.4 Error prevention**: the deliberate Cancel default and the optional type-to-confirm both protect against accidental data loss.

> If you omit `AlertDialogTitle` (very rare), pass `aria-label` on `AlertDialogContent` so screen-reader users still hear the dialog name.

## Responsive behaviour

AlertDialog is centred at every viewport â€” there's no bottom-sheet variant. The panel has `max-w-[calc(100%-2rem)]` so it never overflows on small screens.
