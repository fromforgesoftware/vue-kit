# Dialog

A modal dialog for focused tasks — forms, confirmations, rich previews. Built on Reka UI; content portals to `document.body`. At viewports under 640 px the dialog collapses into a bottom sheet with a drag-to-dismiss handle.

## When to use

- Edit a single object (profile, settings entry, calendar event).
- Walk a user through a short, focused flow that benefits from blocking the page.
- Show rich, scrollable content that doesn't warrant a route change.

## When not to use

- For irreversible destructive actions — use [AlertDialog](?path=/docs/general-alert-dialog--docs) which forces a deliberate Action / Cancel choice and ignores Escape.
- For peripheral help text — use [Tooltip](?path=/docs/general-tooltip--docs) or [Popover](?path=/docs/general-popover--docs).
- For long, multi-step forms — use a route or a [Drawer](?path=/docs/general-drawer--docs).
- For navigation lists — use a Drawer or a side panel.

## Accessibility (WCAG 2.2 AA)

- **2.4.3 Focus order**: focus moves into the dialog on open and is trapped while it is open.
- **2.4.7 Focus visible**: the close button has a visible `:focus-visible` ring.
- **3.2.2 On input**: the dialog only closes on explicit user actions (Escape, overlay click, close button, action button).
- **4.1.2 Name, Role, Value**: the panel has `role="dialog"`, with `aria-labelledby` and `aria-describedby` wired automatically by `DialogTitle` and `DialogDescription`.
- **2.1.2 No keyboard trap**: the focus trap is dismissible via Escape — users are never stuck.
- **2.5.8 Target size**: the close (X) button is 24×24 px (size-6) — meets SC 2.5.8.

> When omitting `DialogTitle` (rare — e.g. a fully visual preview), pass an `aria-label` on `DialogContent` so screen-reader users still hear the dialog name.

## Responsive behaviour

<Canvas of={DialogStories.InteractiveResponsive} />

On screens under 640 px, the dialog renders as a bottom sheet with a drag handle. Dragging the handle down by ≥ 100 px dismisses the sheet.

## Do / Don't

<table>
	<thead>
		<tr>
			<th>Do</th>
			<th>Don't</th>
		</tr>
	</thead>
	<tbody>
		<tr>
			<td>
				Use a clear, action-oriented <code>DialogTitle</code> ("Edit profile", not "Profile").
			</td>
			<td>Open a Dialog from another Dialog — stack thrash and focus chaos.</td>
		</tr>
		<tr>
			<td>Pair every Dialog with a primary action button in the footer.</td>
			<td>Hide essential primary actions inside the body — users scan the footer.</td>
		</tr>
		<tr>
			<td>
				Pick the smallest <code>size</code> that fits the content.
			</td>
			<td>
				Use <code>full</code> as a default — it feels like a bad route.
			</td>
		</tr>
		<tr>
			<td>
				Use <a href="?path=/docs/general-alert-dialog--docs">AlertDialog</a> for irreversible
				destructive actions.
			</td>
			<td>Rely on a Dialog for "are you sure?" — it can be dismissed too easily.</td>
		</tr>
	</tbody>
</table>
