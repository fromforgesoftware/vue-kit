# Drawer

A side panel that slides in from any edge. Supports stacked side-step panels (right / left) for branched workflows. Below 640 px every side renders as a full-screen overlay.

## When to use

- Edit a related entity without losing the parent's context (a row's detail, a calendar event).
- Multi-step or branched flows that benefit from a back stack.
- Filters or actions that fit better on the side than in a Dialog.

## When not to use

- A simple "are you sure" — use [AlertDialog](?path=/docs/general-alert-dialog--docs).
- A focused single-form edit — use [Dialog](?path=/docs/general-dialog--docs).
- Lightweight tooltips or popovers — use [Tooltip](?path=/docs/general-tooltip--docs) or [Popover](?path=/docs/general-popover--docs).

## Stacked side steps

<Canvas of={DrawerStories.TwoLevelSideStep} />

For branched flows (e.g. "configure component" inside a parent form), wrap the second panel in a `DrawerPanel` with a unique `name`, push onto the stack with `DrawerStepTrigger`, and pop with `DrawerStepClose` or `DrawerBack`. The parent panel peeks behind the child so the user keeps context.

Side-step stacking is only available on <code>right</code> and <code>left</code> drawers — top / bottom drawers replace each other.

## Accessibility (WCAG 2.2 AA)

- **2.4.3 Focus order**: panels render with `role="dialog"` and `aria-modal="true"` while active. The drawer is the topmost focusable surface; pushed-back panels get `aria-hidden`.
- **2.4.7 Focus visible**: close, back, and step buttons keep the standard `:focus-visible` ring.
- **2.1.1 Keyboard**: Escape closes the active step (or the whole drawer when at root).
- **4.1.2 Name, Role, Value**: each panel labels itself via `DrawerTitle`. If you omit it, pass `aria-label` on the panel.
- **2.5.8 Target size**: the close button is 24×24 px (size-6) — meets SC 2.5.8.

> Drawers are a focus surface, not a labelled region — give every panel a `DrawerTitle` so screen-reader users know which step they're on.

## Responsive behaviour

<Canvas of={DrawerStories.InteractiveResponsive} />

Below 640 px the drawer fills the screen regardless of side; above 640 px it docks to the chosen edge with a floating, rounded shell.

## Do / Don't

<table>
	<thead>
		<tr>
			<th>Do</th>
			<th>Don't</th>
		</tr>
	</thead>
	<tbody>
		<tr>
			<td>
				Use <code>side="right"</code> for detail panes — the default and most familiar.
			</td>
			<td>Stack three or more side-step panels — flatten the flow.</td>
		</tr>
		<tr>
			<td>
				Pair every panel with a <code>DrawerTitle</code>.
			</td>
			<td>Open Dialogs from inside Drawers — focus thrash.</td>
		</tr>
		<tr>
			<td>
				Use <code>size="md"</code> as the default — the right balance for forms.
			</td>
			<td>
				Use <code>full</code> for everyday use — it feels like a bad route.
			</td>
		</tr>
		<tr>
			<td>Use top / bottom for filters or quick actions.</td>
			<td>Use top / bottom for long forms — landscape phones suffer.</td>
		</tr>
	</tbody>
</table>
