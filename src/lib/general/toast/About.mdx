# Toast

Transient notifications anchored to a corner of the viewport. Mount one `<Toaster />` at the app root, then call the imperative `toast()` / `toast.success()` / `toast.error()` functions from anywhere in your app.

## When to use

- Confirming an asynchronous action ("Saved", "File deleted", "Email sent").
- Surfacing errors from background work the user didn't directly initiate.
- Offering one quick undo for a destructive action via `action`.

## When not to use

- Confirming a successful settings save — see [internal note](#do--dont) below.
- Critical errors that must block until acknowledged — use [Dialog](?path=/docs/general-dialog--docs).
- Inline form validation errors — use field-level error states.
- Long messages — toasts are scanned in passing, not read.

## Anatomy

<Canvas of={ToastStories.Default} />

A single `<Toaster />` lives at the root of your app — typically in `App.vue`. From anywhere else in the app, import `toast` and call it:

```ts
import { toast } from '@fromforgesoftware/vue-kit';

toast.success('Saved', { description: 'Profile updated.' });
toast.error('Could not save', { description: 'Please try again.' });

toast('File deleted', {
	description: 'report-q4.pdf was moved to trash.',
	action: { label: 'Undo', onClick: () => restoreFile() },
});
```

## Accessibility (WCAG 2.2 AA)

- **4.1.3 Status messages**: `Toaster` mounts a `section[aria-live="polite"]`, so screen readers announce new toasts without stealing focus.
- **2.1.1 Keyboard**: action and close buttons are keyboard-operable. The toast itself is reachable via the configurable hotkey (default <kbd>Alt</kbd>+<kbd>T</kbd>).
- **2.2.1 Timing adjustable**: pause-on-hover is built in. Increase `duration` for important messages; consider `Infinity` for errors that need explicit dismissal.
- **2.5.8 Target size**: the close button on a toast measures ≥ 20×20 px. Pair toasts with a destructive `action` label of at least 24×24 px hit area.
- **1.4.3 Contrast**: with `richColors`, toast text and background clear 4.5 : 1.

> Keep error toasts for **background failures** the user couldn't have prevented. Foreground errors (form validation) belong on the field, not in a toast — the user is already looking at the form.

## Responsive behaviour

<Canvas of={ToastStories.InteractiveResponsive} />

The toast stack is fixed-positioned and never causes horizontal overflow. On narrow viewports it should sit near the top or bottom edge — avoid centring on phones because it competes with the keyboard.

## Do / Don't

<table>
	<thead>
		<tr>
			<th>Do</th>
			<th>Don't</th>
		</tr>
	</thead>
	<tbody>
		<tr>
			<td>Confirm async actions ("Email sent", "Schedule published").</td>
			<td>Confirm settings/preferences saves — they're already implied by the form state.</td>
		</tr>
		<tr>
			<td>
				Pair destructive actions with one undo via <code>action</code>.
			</td>
			<td>Stack multiple actions in a single toast — users miss them.</td>
		</tr>
		<tr>
			<td>
				Use <code>toast.error()</code> for background failures.
			</td>
			<td>Use a toast for blocking errors — use a Dialog.</td>
		</tr>
		<tr>
			<td>
				Mount one <code>&lt;Toaster /&gt;</code> per app, at the root.
			</td>
			<td>
				Render <code>&lt;Toaster /&gt;</code> inside multiple layouts — toasts will multiply.
			</td>
		</tr>
		<tr>
			<td>
				Increase <code>duration</code> for messages users need to read in full.
			</td>
			<td>Cram a paragraph into a single toast — write a Dialog or a notification.</td>
		</tr>
	</tbody>
</table>
