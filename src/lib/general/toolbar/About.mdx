# Toolbar

A horizontal or vertical group of related actions with **roving focus** — one tab stop into the toolbar, then arrow keys move between items. Built on Reka UI, implementing the WAI-ARIA Toolbar pattern. Use it for editor formatting bars, view-switcher rows, or any cluster of equally-weighted controls.

## When to use

- Rich-text editor formatting bar (Bold, Italic, Underline, alignment).
- A row of view-switcher actions on top of a list (filter / sort / refresh).
- A group of equally-weighted icon buttons that should share a single tab stop.

## When not to use

- A single primary action — use a [Button](?path=/docs/general-button--docs).
- Form controls — use the form layout patterns.
- Navigation between routes — use [Tabs](?path=/docs/general-tabs--docs) or a Menu.

## Composition

```vue
<Toolbar variant="default" size="default" orientation="horizontal" aria-label="Formatting">
  <ToolbarToggleGroup v-model="format" type="multiple">
    <ToolbarToggleItem value="bold" aria-label="Bold">B</ToolbarToggleItem>
    <ToolbarToggleItem value="italic" aria-label="Italic">I</ToolbarToggleItem>
  </ToolbarToggleGroup>
  <ToolbarSeparator />
  <ToolbarButton>Share</ToolbarButton>
</Toolbar>
```

## Accessibility (WCAG 2.2 AA)

- **1.4.3 Contrast**: item text on `bg-card` clears 4.5 : 1; pressed `data-[state=on]` items use `bg-accent text-accent-foreground` which clears 4.5 : 1.
- **1.4.11 Non-text contrast**: borders, separators, and the focus ring clear 3 : 1.
- **2.1.1 Keyboard**: every interactive item is a real `<button>` or `<a>` — Space, Enter, and arrow keys all work.
- **2.4.3 Focus order**: roving focus — only one item is in the tab order at any moment.
- **2.4.7 Focus visible**: a 3 px ring at `--ring` is applied on `:focus-visible`.
- **2.5.8 Target size**: every item meets the 24×24 px minimum at every size and viewport (default = 36 px, `sm` = 32 px).
- **4.1.2 Name, Role, Value**: the root has `role="toolbar"` (provided by Reka). Always pass an `aria-label` so screen readers can announce the toolbar's purpose. Toggle items expose `aria-pressed` mirroring `data-state`.

## Responsive behaviour

<Canvas of={ToolbarStories.InteractiveResponsive} />

The toolbar uses `min-w-max` so it can scroll horizontally inside an `overflow-x-auto` container on narrow viewports. The container itself never causes layout overflow at the parent level.

## Do / Don't

<table>
	<thead>
		<tr>
			<th>Do</th>
			<th>Don't</th>
		</tr>
	</thead>
	<tbody>
		<tr>
			<td>
				Pass an <code>aria-label</code> on the root toolbar describing its purpose.
			</td>
			<td>Wrap unrelated actions in one Toolbar — group only related controls.</td>
		</tr>
		<tr>
			<td>
				Pair icon-only items with <code>aria-label</code>.
			</td>
			<td>Rely on the icon alone — screen-reader users hear nothing.</td>
		</tr>
		<tr>
			<td>
				Use <code>ToolbarToggleGroup</code> for mutually related toggles (alignment, list type).
			</td>
			<td>Mix toggle and one-shot actions inside the same toggle group.</td>
		</tr>
		<tr>
			<td>Add a separator between unrelated item clusters.</td>
			<td>Add separators decoratively between every item — that's just visual noise.</td>
		</tr>
	</tbody>
</table>
