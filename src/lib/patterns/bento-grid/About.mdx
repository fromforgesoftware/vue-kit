# Bento Grid

A modular grid for laying out heterogeneously-sized cards — the "bento box" pattern popularised by Apple and adopted across modern dashboards. Cards declare a span (`sm`, `md`, `wide`, `tall`, `hero`); the grid uses CSS `grid-auto-flow: dense` to backfill holes left by larger cards with smaller ones that come later, producing a tightly packed visual rhythm without manual line math.

## When to use

- Overview / summary screens with **mixed-importance content** (a hero metric next to many smaller stats).
- Dashboards where **some cards need more horizontal room** (lists, charts) than others (single counts).
- A small-to-medium number of cards (~3–20) with **deterministic order** that matters for accessibility.

## When not to use

- **Variable-height image galleries** — reach for a true masonry instead (CSS `column-count` or a JS library). Bento expects authored sizes, not measured content.
- **Drag-and-drop dashboards** with persistent layouts — use `WidgetGrid`, which adds resize handles and drag affordances.
- **Long uniform feeds** — a regular list or table is simpler and more predictable.

## Dense packing

<Canvas of={BentoGridStories.TallAndDense} />

The grid sets `grid-auto-flow: dense`, which lets the browser **backfill holes** left by larger cards with smaller cards that come later in the DOM. In the "Tall and Dense" story above, a `tall` card followed by `sm`/`md` cards produces a tight layout without any explicit line numbers.

> **A11y note**: `dense` packing changes _visual_ order but **not** DOM / focus order. That's fine when the cards are independent (stats, summaries). Avoid using `dense` when the visual order conveys meaning — e.g. a sequence of steps — because the visual order can diverge from the reading order.

## Hero card

<Canvas of={BentoGridStories.Hero} />

Use `size="hero"` (2×2) sparingly — at most one per grid — for the single most important card on the page.

## Container queries inside cards

`BentoCard` adds the `@container` class so consumers can use Tailwind's container query utilities (`@sm:`, `@md:`) **inside the card** to re-flow content based on the card's actual width. This is the canonical "card adapts to its width" pattern: the grid decides the _card's_ size; the card decides its _interior_ layout.

```vue
<BentoCard size="md">
  <!-- 1 column when narrow, 2 columns when the card is wide -->
  <div class="grid grid-cols-1 @sm:grid-cols-2 gap-2">
    <!-- content -->
  </div>
</BentoCard>
```

## Caller-driven dynamic sizes

When a card's content varies with data (e.g. a Skills card with 2 vs 20 badges), let the **caller** pick the size from props rather than measuring content:

```vue
<BentoCard :size="skills.length > 6 ? 'md' : 'sm'">
  <!-- skill badges -->
</BentoCard>
```

This is deterministic, predictable, and avoids layout flicker. Mature design systems (shadcn, Magic UI, Aceternity) all follow this pattern; none auto-measure content.

## Do / Don't

	<thead>
		<tr>
			<th>Do</th>
			<th>Don't</th>

			</td>
		</tr>
		<tr>
			<td>Use container queries inside cards for interior re-flow</td>
			<td>Try to JS-measure content to auto-resize the card</td>
		</tr>
		<tr>
			<td>
				Drive dynamic sizes from props (<code>:size="n &gt; 6 ? 'md' : 'sm'"</code>)
