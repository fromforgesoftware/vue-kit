# ScheduleTimeline

A multi-employee schedule grid for visualising shift assignments across day, week, and month views. Read-only by design — editing UX is composed in the app layer per-shift via the `#shift-wrapper` slot.

## When to use

- Displaying who works when — one row per employee, one column per day or hour.
- Drilling into a single day at hour-level resolution with overlapping shift lanes.

## When not to use

- Browsing scheduled events for a single calendar — use [`EventCalendar`](?path=/docs/patterns-eventcalendar--docs).
- Editing a single shift's break structure — use [`TimeRangeBar`](?path=/docs/patterns-timerangebar--docs).

## Views

<Canvas of={ScheduleTimelineStories.DayView} />

The day view positions shifts absolutely within an hour grid. The visible window is controlled by `startTime` / `endTime` (partial-hour values are supported). When `stackOverlaps` is enabled, time-overlapping shifts are placed in separate vertical lanes.

<Canvas of={ScheduleTimelineStories.WeekView} />

The week view is a 7-column grid. Off-days render an "OFF" placeholder; multiple shifts on the same day stack vertically as `TimeRangeBar` cards.

<Canvas of={ScheduleTimelineStories.MonthView} />

The month view is a dense calendar grid driven from `selectedDate`'s month (or `dateRange` if provided).

## Grouped vs flat

<Canvas of={ScheduleTimelineStories.WeekView} />

Pass either `departments` (grouped, with collapsible headers) or `rows` (flat). When both are passed, `departments` wins.

<Canvas of={ScheduleTimelineStories.WeekViewGroup} />

The grouped story renders departments with collapsible headers; pass `defaultCollapsed` to start every group collapsed.

## Overlapping shifts

<Canvas of={ScheduleTimelineStories.DayViewBarOverlapping} />

By default, time-overlapping shifts on the same day collapse onto a single lane and render on top of each other. Set `stackOverlaps` (day view only) to spread them into separate vertical lanes using a greedy lane-assignment by start time — the lowest free lane wins.

<Canvas of={ScheduleTimelineStories.DayViewBarOverlappingStacked} />

There is no automatic per-category lane assignment. The previous "schedule / activity / deviation" lane layout has been removed; if you need that grouping, pre-bucket the shifts per row in the consumer and render them via the `#shift-wrapper` slot.

## Props

<Controls of={ScheduleTimelineStories.WeekView} />

<ArgTypes of={ScheduleTimelineStories} />

A few props worth calling out explicitly:

- **`selectedDate: ForgeDate`** — required. Drives the visible week or month, and acts as "today's column" for the day view.
- **`dateRange: { from: ForgeDate; to: ForgeDate }`** — overrides the auto-derived week / month range. Use it to pin a fixed window across navigations.
- **`startTime: string`** — day view only. HH:mm format. Defaults to `'00:00'`. Partial-hour values (e.g. `'06:30'`) are supported.
- **`endTime: string`** — day view only. HH:mm. `'00:00'` is the sentinel for "full 24 hours" (i.e. wraps around to midnight the next day). Defaults to `'00:00'`.
- **`minColumnWidth: number`** — minimum width per column before horizontal scroll kicks in. Default varies by view (day: 80 / mobile 48, week: 140 / mobile 80, month: 36 / mobile 28).
- **`employeeColumnWidth: number`** — width of the sticky employee column. Defaults to 200.
- **`stackOverlaps: boolean`** — day view only. Stacks time-overlapping shifts into separate vertical lanes. Other views ignore this flag.
- **`selectedIds: string[]`** — highlights the matching shift blocks with a primary ring in all three views (day / week / month). Consumers maintain selection state.
- **`rowClass: (row) => ClassValue`** — returns extra classes for each employee row's outer container. All three views honour this (the previous day-view-only restriction has been lifted).
- **`loadMoreThreshold: number`** — opts the timeline into infinite scroll. When set, `load-more` fires once the last virtual row is within this many rows of the end. Omit to disable.
- **`loadKey: string | number`** — opaque token that re-arms the `load-more` latch when the dataset is logically replaced (filter change, sort change, tab swap). Without it, a swap that happens to yield the same `rows.length` won't re-fire `load-more`. Bump this value whenever the underlying dataset's identity changes.
- **`editable: boolean`** — day view only. Enables drag-to-move, edge-resize, and segment drag on each shift card. Emits `shift-resize`, `shift-move`, `shift-segment-update`, and `shift-segment-select` with the shift / employee / date context. Default `false`. See [Edit mode](#edit-mode-day-view) below.
- **`activeSegmentId: string | null`** — forwarded to every editable shift card so an external toolbar can drive segment selection. Pair with `shift-segment-select` to keep the parent-owned state in sync.
- **`selectionMode: 'open' | 'toggle'`** — controls what a plain click on a shift emits. `'open'` (default) fires `shift-click` for the legacy "open detail surface" path; `'toggle'` fires `shift-select(id)` as a single-select replacement. Cmd/Ctrl-click always fires `shift-select` regardless of mode.

## Infinite scroll

<Canvas of={ScheduleTimelineStories.DayViewInfiniteScroll} />
<Canvas of={ScheduleTimelineStories.WeekViewInfiniteScroll} />
<Canvas of={ScheduleTimelineStories.MonthViewInfiniteScroll} />

Set `loadMoreThreshold` to opt in. The timeline emits `load-more` once the user scrolls within that many rows of the end; the parent fetches the next page and appends to `rows`. The component dedupes by tracking the array length at last emission, so a single emission can't fire twice for the same page. The same wiring works across day, week, and month views.

```vue
<ScheduleTimeline
	:rows="rows"
	:selected-date="date"
	view="week"
	:load-more-threshold="hasMore ? 5 : undefined"
	@load-more="fetchNextPage"
/>
```

Disable by omitting `loadMoreThreshold` (or setting it to `undefined`) once `hasMore` is false — that keeps the virtualizer's watcher cheap and prevents a redundant emission at the very end of the list.

```vue
<!-- Reset the latch when the filter changes by bumping loadKey. -->
<ScheduleTimeline
	:rows="rows"
	:selected-date="date"
	view="week"
	:load-more-threshold="hasMore ? 5 : undefined"
	:load-key="`${filterId}|${sortKey}`"
	@load-more="fetchNextPage"
/>
```

## Edit mode (day view)

<Canvas of={ScheduleTimelineStories.DayViewEditable} />

Set `editable` to put the day-view shift cards into edit mode. Three gestures become available on each card:

- **Drag the card body along the hour axis → `shift-move`.** The move is implemented at the day-shift wrapper level (not inside the inner `TimeRangeBar`) so the drag math uses the parent timeline's `columnWidth` — the hour-grid px-per-hour — and the card visually translates along the day's hour columns during drag. On commit, the wrapper emits `shift-move` with the new wall-clock `startTimeSec`; the duration is unchanged. `TimeRangeBar` deliberately has no "move whole bar" gesture — see its docs for the rationale.
- **Drag the left or right edge → `shift-resize`.** The inner `TimeRangeBar` handles this via its `resizable` mode and emits `RangeResizeEvent`; the wrapper translates the bar's zero-anchored `startTimeSec` to wall-clock seconds before re-emitting.
- **Drag a child segment (break / lunch / sub-status) → `shift-segment-update`.** Plain-click a segment → `shift-segment-select` (parent-owned selection). Pass `activeSegmentId` on `ScheduleTimeline` to drive it from an external toolbar.

Each of the four edit events carries `{ shiftId, employeeId, dateStr, … }` so a flat handler on the page can route mutations into the right draft without re-deriving context.

Week and month views ignore `editable` — those densities have no time axis for drag-resize. Use `selectionMode` (below) to enable selection in those views.

The `#shift-wrapper` slot scope exposes `editable` and `activeSegmentId`, so consumers that render their own shift content can branch on edit state without re-reading the timeline's props.

## Selection mode

`selectionMode` decouples "click a shift" from "open a detail surface":

- **`'open'`** (default) — plain click emits `shift-click(payload)`; Cmd/Ctrl-click emits `shift-select(id)` for multi-toggle. Matches the legacy behaviour of the read-only timeline.
- **`'toggle'`** — plain click emits `shift-select(id)` as a single-select replacement; Cmd/Ctrl-click still emits `shift-select` for multi-toggle. `shift-click` is not emitted in this mode.

The Cmd/Ctrl chord is preserved across modes — the keyboard idiom for "extend my selection" should never change shape. Use `'toggle'` on edit pages so a click selects a shift for an external toolbar instead of opening a detail panel.

## Empty state

<Canvas of={ScheduleTimelineStories.EmptyStateWithSlot} />

Pass `#empty` whenever the rows array can be empty for reasons the user can act on (a filter that excluded everyone, a group with no members, a search miss). The slot's container fills the rows area, so a centred message reads naturally inside the timeline frame. The sticky header keeps the date or hour scale on screen so the user retains spatial context.

Compose with the shared [`EmptyState`](?path=/docs/general-emptystate--docs) component — the same one [`DataTable`](?path=/docs/patterns-datatable--docs) uses for "no results" — so the visual language stays consistent across the app. Add `class="h-full"` so it fills the available rows area.

```vue
<ScheduleTimeline :rows="rows" :selected-date="date" view="week">
  <template #empty>
    <EmptyState
      :icon="UsersRound"
      title="No employees match the current filter"
      description="Adjust the filter to show employees."
      size="sm"
      class="h-full"
    />
  </template>
</ScheduleTimeline>
```

## Employee status

Pass `employee.status` to render the small dot on the avatar:

- `'online'` — success green
- `'offline'` — muted neutral
- omit — no dot

The previous `statusColor: string` field has been replaced by this typed union — callers that were passing `'green'` should switch to `'online'`.

## How to compose editing

The timeline is intentionally read-only — it owns layout and selection visuals, not edit semantics. To add editing, use the `#shift-wrapper` scoped slot: the slot scope exposes the shift, its sibling shifts, the row's `employeeId`, and the cell's `dateStr` (plus the layout payload — `style`, `dataAttrs`, `ariaLabel`, `selected`). Inside the slot, render whatever editing UI the page needs. The timeline keeps positioning the wrapper for you; you keep ownership of the per-shift content.

A typical pattern wraps each shift in an editable `<TimeRangeBar>` (for break / lunch authoring) or a `<ShiftBar>` (for shift-level move / resize):

```vue
<ScheduleTimeline :rows="rows" :selected-date="date" view="week">
  <template #shift-wrapper="{ shift, employeeId, dateStr, style, dataAttrs }">
    <div :style="style" v-bind="dataAttrs">
      <TimeRangeBar
        editable
        :duration-sec="shift.durationSec"
        :start-time-sec="parseHHMMToSec(shift.time)"
        :segments="toSegments(shift.children)"
        @segment-update="onChildUpdate(shift.id, $event)"
        @range-resize="onShiftResize(shift.id, employeeId, dateStr, $event)"
      />
    </div>
  </template>
</ScheduleTimeline>
```

The slot fires for every shift in every view, so the same composition reads consistently across day, week, and month.
